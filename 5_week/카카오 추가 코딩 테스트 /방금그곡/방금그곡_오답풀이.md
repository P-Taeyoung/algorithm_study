## 1. `#` 음 처리의 결정적 패착

- **실수했던 점:** 처음에는 `#`을 별개의 인덱스로 취급하거나, `while` 루프 내에서 `if sheet[idx] == "#"` 식으로 실시간 처리를 시도했습니다. 이로 인해 인덱스 계산이 꼬이고 비교 로직이 매우 복잡해졌습니다.
- **깨달은 점:** `#`이 붙은 음(`C#`)은 무조건 **한 단위**로 묶어야 합니다.
  - ✅ **해결책:** 문자열을 미리 리스트로 변환(`['A', 'B', 'C#']`)하거나, 특정 단일 문자(`C#` → `c`)로 **치환(Replace)** 하여 길이를 1로 맞춘 뒤 계산하는 것이 가장 안전합니다.

## 2. 재생 시간(Full Sheet) 생성 로직

- **실수했던 점:** `while`문과 `%` 연산자를 써서 실시간으로 악보를 생성하려다 보니, `#` 위치에 따라 `cnt`는 안 늘고 `idx`만 늘어나는 등 복잡한 예외 상황이 발생했습니다.
- **깨달은 점:** 원본 악보를 먼저 리스트화한 뒤, 파이썬의 리스트 연산을 활용하면 **한 줄**로 해결됩니다.
  - ✅ **해결책:** `(리스트 * 반복횟수)[:재생시간]` 패턴을 사용하세요. 어떤 상황에서도 정확한 연주 길이를 보장합니다.

```python
full_sheet = (origin_sheet_list * (play_time // len(origin_sheet_list) + 1))[:play_time]
```

## 3. 비교 로직의 정합성 (Partial Match 방지)

- **실수했던 점:** `while`문으로 인덱스를 하나씩 비교할 때, 루프가 끝났다고 해서 무조건 '일치'라고 판단했습니다.
- **깨달은 점:** 멜로디의 끝까지 다 확인했는지(`m_idx == len(m_sheet)`)를 **반드시 체크**해야 합니다. 노래 뒷부분이 짧아서 루프가 끝난 경우를 걸러내야 하기 때문입니다.
  - ✅ **해결책:** 파이썬의 **리스트 슬라이싱 비교**를 사용하면 훨씬 직관적이고 안전합니다.

```python
if full_sheet[j : j + len(m_sheet)] == m_sheet:
```

## 4. 우선순위 및 초기값 설정

- **실수했던 점:** `max_play_time = 0`으로 설정하여 재생 시간이 0분인 경우 처리가 모호해질 수 있었고, 재생 시간이 같을 때의 우선순위 처리가 미흡했습니다.
- **깨달은 점:** 문제의 조건인 **"시간이 길 것 > 같다면 먼저 입력된 것"** 을 명확히 구현해야 합니다.
  - ✅ **해결책:**
    1. `max_play_time = -1`로 초기화.
    2. `if current_time > max_play_time:` 조건을 사용하면, 시간이 같을 때는 `>` 조건이 거짓이 되어 자연스럽게 먼저 입력된 곡이 유지됩니다.
    3. 혹은 `enumerate`를 사용해 입력 순서(`idx`)를 보관한 뒤 `sort(key=lambda x: (-time, idx))`로 정렬합니다.

## 💡 알고리즘 꿀팁 요약

| 항목 | 기존 방식 (위험) | 개선 방식 (추천) |
|------|----------------|----------------|
| 시간 계산 | `:` 제거 후 숫자 뺄셈 | `(H * 60) + M` 분 단위 통일 |
| `#` 음 처리 | 인덱스 기반 조건문 분기 | 리스트 변환 또는 소문자 치환 |
| 반복 악보 | `while` + `%` 연산자 | 리스트 곱셈(`*`) + 슬라이싱(`[:]`) |
| 다중 조건 정렬 | `if`문 중첩 관리 | `enumerate` + `sort(key=...)` |

## 🗝️ 핵심 3단계 기억법

비슷한 **문자열 시뮬레이션** 문제를 만나면 이 세 가지만 기억하세요:

1. **전처리** — 치환 또는 리스트화로 단위를 통일
2. **단위 통일** — 시간, 인덱스 등 기준을 하나로 맞추기
3. **슬라이싱 활용** — 복잡한 루프 대신 파이썬다운 비교

> 이 문제는 단순 구현력이 아니라 **"데이터를 얼마나 다루기 좋게 가공하느냐(Preprocessing)"** 가 핵심입니다.
> 특히 `#` 처리를 위해 리스트로 묶어버리는 접근이 문제 풀이의 열쇠였습니다.

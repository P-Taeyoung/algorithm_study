# 🔍 이분 탐색(Binary Search), 언제 써야 할까?

> 알고리즘을 공부할 때 특정 기법을 **'어떻게'** 구현하는지 아는 것도 중요하지만,  
> **'언제'** 꺼내 써야 하는지(사용 조건과 시그널)를 아는 것이 진짜 실력입니다.

---

## 🚨 시그널 1. 탐색 범위($N$)가 비정상적으로 클 때 (가장 강력한 힌트)

문제의 제한 사항을 봤는데 탐색해야 하는 범위나 데이터의 개수가 무지막지하게 크다면 **가장 먼저 이분 탐색을 의심**해야 합니다.

- **기준**: 데이터의 크기($N$)가 10만($10^5$) ~ 100만을 넘어가거나, 탐색 범위가 10억($10^9$) 단위일 때
- **이유**: $O(N)$ 알고리즘으로는 무조건 시간 초과(TLE)가 납니다. 탐색 범위를 절반씩 날려버리는 $O(\log N)$ 알고리즘이 필수적입니다.

| 알고리즘 | 시간 복잡도 | $N = 10^6$ 기준 연산 횟수 |
|----------|------------|--------------------------|
| 선형 탐색 | $O(N)$ | 1,000,000번 ❌ |
| 이분 탐색 | $O(\log N)$ | 약 20번 ✅ |

## 🚨 시그널 2. "특정 조건 이상/이하의 개수"를 찾을 때 (이번 카카오 문제!)

어떤 값을 **정확히** 찾는 게 아니라, 조건을 만족하는 **경계선(Boundary)** 을 찾을 때 매우 유용합니다.  
이를 `Lower Bound` / `Upper Bound` 라고 부릅니다.

- **키워드**: "X점 **이상**인 사람은 몇 명인가?", "무게가 Y **이하**인 화물의 개수는?"
- **이유**: 배열이 정렬되어 있다면, X점 이상이 '시작되는 첫 번째 위치(`bisect_left`)'만 찾으면 그 뒤는 볼 필요도 없이 전부 조건을 만족하기 때문입니다.

```python
from bisect import bisect_left

scores = [30, 50, 60, 70, 80, 90]  # 정렬된 상태
target = 60

# 60점 이상인 사람의 수
idx = bisect_left(scores, target)
count = len(scores) - idx  # → 4명
```

## 🚨 시그널 3. "최댓값/최솟값 구하기" → "가능한가? (Yes/No)"로 바꿀 수 있을 때

이분 탐색의 가장 멋진 응용 분야인 **파라메트릭 서치(Parametric Search)** 입니다.  
코딩 테스트 단골 손님입니다.

- **키워드**: "최댓값을 구하시오", "최솟값을 구하시오" (특히 "가장 가까운 거리의 최댓값", "시간의 최솟값" 등)

### 변환 방법

| 원래 질문 | 이분 탐색용 질문으로 변환 |
|-----------|--------------------------|
| "공유기 3대를 설치할 때, 인접한 두 공유기 사이 거리의 **최댓값**은?" | "거리를 X로 고정했을 때, 공유기 3대를 설치하는 것이 **가능한가(Yes/No)?**" |

### 동작 원리

```
거리 X를 이분 탐색으로 조절 ("업다운 게임")
  ├── 설치 가능(Yes) → 거리를 더 늘려봄 (Up)
  └── 설치 불가(No)  → 거리를 줄임 (Down)
```

> 이처럼 **"최적화 문제"** 를 **"결정 문제(Yes/No)"** 로 바꿀 수 있다면 파라메트릭 서치를 적용할 수 있습니다.

## 🚨 시그널 4. 데이터는 고정, 질문(Query)만 엄청 많을 때

이번 카카오 문제와 **정확히 일치**하는 상황입니다.

- **상황**: 정보(Info)는 한 번 주어지고 고정되어 있는데, 질문(Query)이 수만 ~ 수십만 개 쏟아질 때
- **이유**: 데이터가 고정되어 있다면 처음에 **딱 한 번만** 정렬($O(N \log N)$)해 두면, 이후의 수많은 질문들을 이분 탐색($O(\log N)$)으로 순식간에 처리할 수 있어 **가성비**가 엄청납니다.

```
[전처리] 딱 한 번 정렬: O(N log N)
[반복]   질문 Q개 처리: O(Q × log N)

vs.

[반복]   질문마다 선형 탐색: O(Q × N) ← TLE 💀
```

## ⚠️ 절대 잊지 말아야 할 대전제 (Rule 0)

> 이 모든 시그널이 켜졌더라도, 이분 탐색을 쓰기 위한 **절대적인 전제 조건**이 하나 있습니다.

$$\boxed{\text{데이터가 반드시 정렬(Sort)되어 있거나, 정렬할 수 있는 상태여야 한다.}}$$

값의 흐름이 오름차순/내림차순 등 **일정한 방향성(단조성, Monotonicity)** 을 가져야 합니다.  
정렬로 인해 원래의 인덱스나 순서 정보가 훼손되어 문제가 꼬이는 경우가 아니라면, 위 4가지 시그널을 만났을 때 자신 있게 이분 탐색(또는 `bisect`)을 꺼내 드세요.

## 📋 시그널 요약표

| 시그널 | 핵심 키워드 | 기법 |
|--------|------------|------|
| 1️⃣ 탐색 범위가 비정상적으로 클 때 | $N \geq 10^5$, 범위 $\geq 10^9$ | 기본 이분 탐색 |
| 2️⃣ 조건 이상/이하의 개수를 찾을 때 | "몇 명?", "몇 개?" | `bisect_left` / `bisect_right` |
| 3️⃣ 최적값을 Yes/No로 바꿀 수 있을 때 | "최댓값", "최솟값" | 파라메트릭 서치 |
| 4️⃣ 데이터 고정 + 질문이 많을 때 | Info 고정, Query 다량 | 전처리 정렬 + 이분 탐색 |

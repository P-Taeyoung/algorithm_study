# 📝 알고리즘 오답노트: 도넛과 막대 그래프 (그래프 차수 분석)

---

## 1. 문제 유형의 특성: "탐색의 함정"

- **착각하기 쉬운 포인트**: '그래프', '연결', '모양'이라는 키워드 때문에 자연스럽게 DFS나 BFS를 떠올리기 쉽다.
- **실제 요구 역량**: 그래프의 위상적 특징(정점의 차수, In-degree / Out-degree) 파악.
- **퍼포먼스 고려사항**: 간선의 길이(`edges`)가 최대 1,000,000이다. 만약 모양을 판별하기 위해 직접 탐색을 돌린다면, 매 탐색마다 `visited` 배열을 생성하고 순환을 체크하는 과정에서 막대한 시간과 메모리 오버헤드가 발생한다. 따라서 $O(E)$의 시간 복잡도로 한 번의 순회만으로 답을 도출해야 한다.

> **⚠️ 왜 DFS/BFS가 여기서 독인가?**
> 탐색 기반 풀이는 "어떤 정점들이 연결되어 있는가"를 파악하는 데 강점이 있지만,
> 이 문제는 이미 연결 구조가 주어진 상태에서 **구조의 형태**를 분류하는 문제다.
> 즉, 탐색이 아니라 **분류(Classification)** 문제이며, 분류의 기준은 각 정점이 가진 차수(degree) 값이다.

---

## 2. 올바른 문제 접근 프레임워크

### 🔍 Step 0. 문제를 읽을 때 던져야 할 질문

> *"이 문제가 진짜로 요구하는 것이 '탐색'인가, 아니면 '특성 파악'인가?"*

문제에 다음과 같은 신호가 보이면 탐색보다 **차수 분석(Degree Analysis)** 을 먼저 고려해야 한다.

| 신호 | 의미 |
|---|---|
| 그래프의 "모양"이나 "종류"를 판별하라 | 구조적 특징(차수)으로 분류 가능 |
| 간선 수가 매우 크다 (10⁵ ~ 10⁶) | $O(E)$ 단일 순회가 필수 |
| 연결 컴포넌트 전체를 탐색할 필요가 없다 | 특정 정점의 국소적 특성만으로 충분 |
| "생성된 정점"처럼 인위적으로 추가된 노드가 존재 | 해당 노드의 차수가 핵심 단서 |

### 🧩 Step 1. 각 그래프 형태의 위상적 불변량(Topological Invariant) 도출

알고리즘 문제에서 그래프의 형태를 판별할 때는 **모든 형태가 반드시 가지는 유일한 구조적 특성**을 먼저 정리해야 한다.
이 문제에서 각 형태별 불변량은 다음과 같다.

```
방향 그래프(Directed Graph) 기준

도넛 모양  :  모든 정점의 in-degree = 1, out-degree = 1
             → 순환 고리이므로 특이 정점 없음

막대 모양  :  시작 정점 out-degree = 1, in-degree = 0 (중간 정점들은 1:1)
             → 끝 정점의 out-degree = 0 이 반드시 1개 존재

8자 모양   :  두 고리가 만나는 교차점 정점 1개
             → 해당 정점의 out-degree = 2, in-degree = 2
```

> 💡 핵심 통찰: **도넛은 특이 정점이 없기 때문에** 직접 카운트가 불가능하다.
> 막대와 8자만 직접 세고, 나머지를 전체에서 빼는 **간접 도출 전략**이 필요하다.

### 🏗️ Step 2. 생성된 정점(루트 노드)의 역할 파악

이 문제에는 모든 그래프를 연결하기 위해 인위적으로 추가된 **생성 정점**이 존재한다.
이 정점의 특성을 먼저 파악하는 것이 풀이의 출발점이다.

```
생성 정점의 특성:
  - 어떤 그래프에서도 들어오는 간선이 없다  →  in-degree = 0
  - 각 그래프의 임의 정점으로 간선을 내보낸다  →  out-degree ≥ 2

따라서:
  생성 정점 = (in-degree == 0) AND (out-degree >= 2) 를 만족하는 정점
  전체 그래프 수 = 생성 정점의 out-degree 합산
```

> **왜 out-degree ≥ 2인가?**
> out-degree = 1이면 막대 그래프의 일반 중간 정점과 구분이 불가능하다.
> 문제 조건상 생성 정점은 2개 이상의 그래프와 연결되므로 반드시 2 이상이다.

### 📐 Step 3. 판별 순서 설계 (의존 관계 고려)

카운팅 순서에도 **의존 관계**가 있다. 잘못된 순서로 집계하면 오답이 나온다.

```
올바른 판별 순서:

① 생성 정점 식별  →  전체 그래프 수(N) 확정
② 막대 그래프 수  →  out-degree = 0인 정점 개수 카운트
③ 8자 그래프 수   →  out-degree = 2 AND in-degree = 2인 정점 개수 카운트
④ 도넛 그래프 수  →  N - (막대 수 + 8자 수)
```

> ⚠️ **주의**: 생성 정점 자신의 차수가 막대/8자 조건에 걸리지 않도록
> 집계 시 생성 정점은 반드시 제외해야 한다.

## 3. 핵심 접근 방식 (Strategy) 요약

그래프를 직접 순회하지 않고, 각 그래프 형태가 가지는 **유일하고 뚜렷한 특징점(특이 정점)의 차수**를 세어 판별한다.

1. **간선 정보 카운팅**: 모든 간선을 순회하며 각 정점별로 `[나가는 간선 수, 들어오는 간선 수]`를 기록한다.
2. **정점별 특징 매칭**:

| 정점 유형 | 조건 | 비고 |
|---|---|---|
| **생성된 정점** (무관한 정점) | out-degree ≥ 2, in-degree = 0 | 이 정점의 out-degree 합 = **전체 그래프 수** |
| **막대 모양** 그래프 | out-degree = 0인 정점이 반드시 1개 존재 | 끝 정점 탐지 |
| **8자 모양** 그래프 | out-degree = 2, in-degree = 2인 중심 정점이 반드시 1개 존재 | 교차점 탐지 |
| **도넛 모양** 그래프 | 특징적인 정점 없음 | `전체 - (막대 + 8자)`로 간접 도출 |

## 4. 코드 최적화: `collections.defaultdict`의 활용

### ❌ AS-IS: 비효율적인 분기 처리

```python
if from_node not in linked_nodes.keys():
    linked_nodes[from_node] = [0, 0]
    linked_nodes[from_node][0] += 1
else:
    linked_nodes[from_node][0] += 1
```

**문제점**: 매 간선마다 키 존재 여부를 확인하기 위해 `.keys()`를 호출하거나 `if-else` 분기를 타야 하므로 코드가 길어지고, 대규모 데이터 처리 시 미세한 성능 저하가 발생할 수 있다.

### ✅ TO-BE: `defaultdict`를 활용한 개선

```python
from collections import defaultdict

# 키가 없을 경우 자동으로 [0, 0]을 할당하는 딕셔너리 생성
linked_nodes = defaultdict(lambda: [0, 0])

for a, b in edges:
    linked_nodes[a][0] += 1  # a에서 나가는 간선 (Out-degree) 증가
    linked_nodes[b][1] += 1  # b로 들어오는 간선 (In-degree) 증가
```

**개선 효과**:
- `KeyError`를 방지하기 위한 보일러플레이트 코드가 완벽히 제거된다.
- 진입점(`In`)과 진출점(`Out`)의 상태를 기록하는 핵심 비즈니스 로직에만 집중할 수 있어 코드 가독성이 매우 높아진다.

### ✅ 전체 풀이 구조 (pseudo-code)

```python
from collections import defaultdict

def solution(edges):
    degree = defaultdict(lambda: [0, 0])  # [out, in]

    for a, b in edges:
        degree[a][0] += 1
        degree[b][1] += 1

    # ① 생성 정점 탐지 및 전체 그래프 수 계산
    total, created_node = 0, -1
    for node, (out, inp) in degree.items():
        if out >= 2 and inp == 0:
            total = out
            created_node = node
            break

    # ② 막대 / 8자 그래프 수 카운트 (생성 정점 제외)
    bar, eight = 0, 0
    for node, (out, inp) in degree.items():
        if node == created_node:
            continue
        if out == 0:
            bar += 1
        if out == 2 and inp == 2:
            eight += 1

    # ③ 도넛 그래프 수 = 전체 - (막대 + 8자)
    donut = total - bar - eight

    return [created_node, donut, bar, eight]
```

## 💡 회고 (Retrospective)

### 핵심 교훈

> 규모가 큰 데이터를 다룰 때는 습관적으로 탐색 배열을 생성하기 전에,
> **데이터가 가진 구조적 특징(상태값)만으로 문제를 해결할 수 없는지 먼저 고민할 것.**

### 사고 패턴 개선 체크리스트

문제를 처음 읽었을 때 아래 순서로 사고하는 습관을 들이자.

- [ ] **제약 조건부터 확인**: 데이터 크기가 크면 $O(N \log N)$ 이하 알고리즘이 필요하다는 신호
- [ ] **"직접 탐색이 꼭 필요한가?" 자문**: 연결 구조를 알아야 하는가, 아니면 형태만 판별하면 되는가
- [ ] **각 요소의 불변량 도출**: 모든 경우에 성립하는 수학적/구조적 특성을 먼저 정리
- [ ] **간접 도출 가능성 검토**: 직접 구하기 어려운 값은 전체에서 나머지를 빼서 구할 수 없는가
- [ ] **의존 관계 정리**: 어떤 값을 먼저 구해야 다음 값을 구할 수 있는지 순서 설계

### 유사 문제 유형 (같은 사고방식 적용 가능)

| 문제 패턴 | 핵심 접근 |
|---|---|
| 트리에서 리프 노드 찾기 | degree = 1인 정점 카운트 |
| 위상 정렬 (Topological Sort) | in-degree = 0인 정점부터 처리 |
| 이분 그래프 판별 | 홀수 사이클 존재 여부 (차수 패리티로 판별 가능한 경우) |
| 오일러 경로/회로 판별 | 홀수 차수 정점의 개수로 판별 |

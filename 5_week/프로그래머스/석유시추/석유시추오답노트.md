# 📝 [종합 오답 노트] 프로그래머스 - 석유 시추 (BFS)

---

## 1. 🎯 문제 접근 방식 (나의 초기 아이디어)

| 항목 | 내용 |
|------|------|
| **핵심 로직** | BFS를 활용해 석유가 있는 구역(덩어리)을 탐색하고, 각 구역마다 고유한 ID(번호)를 부여한다. |
| **중복 방지** | 나중에 열(Column)을 기준으로 석유량을 합산할 때, 같은 구역을 중복해서 더하지 않도록 `Set`과 `Dictionary`를 활용해 방문한 구역을 체크한다. |
| **결과** | 아이디어의 방향성은 완벽하여 정확성 테스트는 모두 통과. 하지만 효율성 테스트에서 **시간 초과(반타작) 발생**. |

---

## 2. 🚨 효율성 실패의 3가지 핵심 원인 (Why TLE?)

### ① 가장 치명적인 실수: 2차원 배열의 반복 생성 (Overhead)

- 석유 덩어리를 발견할 때마다 `get_all_oil` 함수 내부에서 `visited = [[False] * M for _ in range(N)]`를 **새로 생성**함.
- 파이썬에서 거대한 2차원 배열을 메모리에 할당하고 초기화하는 작업은 그 자체로 $O(N \times M)$의 시간 복잡도를 가짐.
- 탐색을 시작하기도 전에 배열을 만드는 데에만 막대한 시간과 메모리를 낭비하여 **병목 현상의 주원인**이 됨.

### ② 좌표 기록 방식의 비효율 (Dictionary vs 2D Array)

- 탐색한 좌표가 몇 번 구역인지 기록하기 위해 `dic[(new_r, new_c)] = n` 형태로 **딕셔너리**를 사용함.
- 좌표를 튜플로 묶고 **해싱(Hashing)하는 과정**이 추가되어, `맵(land)`과 동일한 크기의 2차원 배열의 인덱스로 직접 접근하는 것보다 미세하게 속도가 느림.

### ③ 시작점 누락 및 혼재된 로직

- 시작점 `(r, c)`를 딕셔너리에 추가하지 않는 **작은 논리적 허점**이 있었음.
- 탐색(BFS)을 하다가 계산을 하고, 다시 탐색을 하는 식으로 **로직이 섞여** 있어 구조가 다소 복잡했음.

---

## 3. 💡 해결책: 2-Phase (탐색과 계산의 분리) 전략

문제를 가장 효율적이고 깔끔하게 풀기 위해 로직을 두 단계로 완전히 분리함.

```
[Phase 1] 탐색 전담          [Phase 2] 계산 전담
─────────────────────        ──────────────────────────────
맵 전체를 한 번 순회 →       열(Column) 단위 순회 →
BFS로 덩어리 탐색 →          chunk_map에서 구역 ID 확인 →
chunk_map에 ID 라벨링 →      미리 구해둔 석유량 합산 →
area_sum_oil에 총량 저장      최댓값 갱신
```

- **Phase 1 (탐색 전담):** 무조건 맵 전체를 한 번 훑으며 모든 석유 덩어리를 파악한다.  
  → 이때 전역 `visited` 배열 1개, 구역을 기록할 `chunk_map` 배열 1개만 **최상단에서 딱 한 번 생성**하여 계속 재사용.

- **Phase 2 (계산 전담):** 열(Column) 단위로 순회하며, 석유(1)가 보이면 `chunk_map`에서 구역 ID를 확인하고 미리 구해둔 석유량을 더해 최댓값을 갱신한다.

## 4. 🔍 코드 비교 (Before & After)

### ❌ AS-IS: 시간 초과가 발생한 초기 접근

```python
# 탐색할 때마다 배열을 매번 새로 생성함 (시간 초과 원인)
def get_all_oil(land, r, c, dic, n):
    visited = [[False] * len(land[0]) for _ in range(len(land))]  # ← 💀 매번 O(N×M) 초기화!
    # ... 중략 ...
    dic[(new_r, new_c)] = n  # ← 튜플을 키로 하는 딕셔너리 사용 (해싱 오버헤드)
```

**문제점 요약:**

| 문제 | 원인 | 영향 |
|------|------|------|
| 배열 반복 생성 | 함수 내부에서 `visited` 매번 초기화 | $O(N \times M)$ × 덩어리 수 만큼 낭비 |
| 딕셔너리 좌표 저장 | 튜플 해싱 오버헤드 | 2차원 배열 인덱스 접근보다 느림 |
| 시작점 누락 | `(r, c)` 미등록 | 논리적 오류 가능성 |
| 로직 혼재 | 탐색 중 계산 수행 | 코드 복잡도 증가 |

---

### ✅ TO-BE: 효율성을 극대화한 최적화 코드

```python
from collections import deque

def solution(land):
    R, C = len(land), len(land[0])
    
    # 💡 [핵심] 배열 생성은 최상단에서 딱 한 번만!
    visited = [[False] * C for _ in range(R)]
    chunk_map = [[0] * C for _ in range(R)]  # dic 대신 2차원 배열 사용 (속도 UP)
    area_sum_oil = {}
    
    n = 1  # 구역 ID

    # ─── [Phase 1] 전체 맵 탐색 및 구역 라벨링 ───────────────────────────
    for i in range(R):
        for j in range(C):
            if land[i][j] == 1 and not visited[i][j]:
                dq = deque([(i, j)])
                visited[i][j] = True
                chunk_map[i][j] = n  # 시작점 기록 (누락 방지!)
                sum_oil = 1
                
                while dq:
                    cur_r, cur_c = dq.popleft()
                    for dr, dc in [(-1, 0), (1, 0), (0, -1), (0, 1)]:
                        new_r, new_c = cur_r + dr, cur_c + dc
                        if 0 <= new_r < R and 0 <= new_c < C:
                            if land[new_r][new_c] == 1 and not visited[new_r][new_c]:
                                visited[new_r][new_c] = True
                                chunk_map[new_r][new_c] = n  # 인덱스 직접 접근 (빠름!)
                                sum_oil += 1
                                dq.append((new_r, new_c))
                
                area_sum_oil[n] = sum_oil
                n += 1

    # ─── [Phase 2] 열(Column)별로 순회하며 최대 석유량 계산 ─────────────
    max_sum_oil = 0
    for c in range(C):
        visited_area = set()
        cur_sum_oil = 0
        
        for r in range(R):
            if land[r][c] == 1:
                area_id = chunk_map[r][c]
                if area_id not in visited_area:
                    visited_area.add(area_id)  # 이번 열에서 이미 더한 구역 체크
                    cur_sum_oil += area_sum_oil[area_id]
        
        max_sum_oil = max(max_sum_oil, cur_sum_oil)

    return max_sum_oil
```

**개선점 요약:**

| 항목 | AS-IS (❌) | TO-BE (✅) |
|------|-----------|-----------|
| `visited` 배열 생성 | 탐색마다 반복 생성 | **최상단에서 1회만 생성** |
| 좌표 저장 방식 | `dict[(r, c)]` (해싱) | `chunk_map[r][c]` (**인덱스 직접 접근**) |
| 시작점 처리 | 미등록 (버그 위험) | **명시적으로 등록** |
| 탐색/계산 구조 | 혼재 | **Phase 1 / Phase 2 완전 분리** |
| 시간 복잡도 | $O(K \times N \times M)$ | **$O(N \times M)$** |

> $K$ = 석유 덩어리 수

## 5. 🌟 오늘의 핵심 교훈 (Key Takeaway)

> **"파이썬에서 큰 자료구조(배열)를 만드는 것은 공짜가 아니다!"**

```
탐색 루프 안에서 O(N×M) 배열을 반복 생성
        ↓
무조건 시간 초과 (TLE)
```

### ✏️ 앞으로 지킬 3가지 원칙

1. **크기가 큰 2차원 배열은 반드시 최상단(전역)에서 한 번만 생성하고, 값을 덮어씌우며 재사용한다.**
2. **좌표를 매핑할 때는 딕셔너리보다 동일한 크기의 2차원 배열을 활용한다.** (인덱스 접근이 해싱보다 빠름)
3. **탐색 로직과 계산 로직은 Phase를 나눠 완전히 분리한다.** (코드 가독성 + 효율성 동시 확보)

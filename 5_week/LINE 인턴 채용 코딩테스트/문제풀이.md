# 📝 코딩테스트 핵심 분석 리포트: 움직이는 타겟 잡기 (BFS)

## 1. 초기 풀이 분석: 1차원 전수 시뮬레이션

유저님이 최초로 고안하신 `catch_me` 함수와 `catch_me_1` 함수의 핵심 접근 방식입니다.

### 💡 풀이 논리
매 초(시간)마다 브라운이 이동할 수 있는 모든 위치를 큐(Queue) 또는 배열에 담아 기록하고, 코니가 해당 초에 도착한 위치와 일치하는지 비교하는 정공법입니다.

### 🟢 장점 (정확성)
* 문제의 요구사항인 **"시간의 흐름에 따른 상태 변화"**를 가장 직관적으로 완벽하게 시뮬레이션했습니다.
* `for _ in range(len(queue)):` 등을 활용하여 큐를 레벨(시간) 단위로 끊어서 탐색하는 정석적인 BFS 패턴을 잘 적용했습니다.

### 🔴 한계점 (시간 초과)
* 브라운은 `+1`, `-1` 이동을 통해 이전 위치로 되돌아가는 '제자리걸음(와리가리)'이 가능합니다. (ex. `10 -> 11 -> 10`)
* 이 방식을 사용하면 이미 방문했던 위치라도 다음 초, 다다음 초에 **계속해서 큐에 중복으로 들어가게 됩니다.**
* 코니가 멀리 도망갈수록 연산량이 $O(3^N)$으로 기하급수적으로 폭발하여 **시간 초과(TLE)** 및 메모리 초과가 발생합니다.

## 2. 최적화 풀이: 상태 공간 확장 (홀짝 Parity BFS)

초기 풀이의 '상태 폭발'을 막기 위해 **가지치기(Pruning)**를 얹은 고도화된 최적화 기법입니다.

### 💡 핵심 원리
브라운이 특정 위치 $X$에 **2초(짝수)**에 도착했다면, 1초 앞뒤로 움직여서 **4초, 6초, 8초...** 에도 언제든 $X$에 다시 올 수 있습니다. 반대로 홀수 초에 도착했다면 홀수 초마다 올 수 있습니다.

### 🚀 최적화 방법 (가지치기)
* 매 초 모든 위치를 큐에 넣는 대신, 각 위치마다 **"최초로 도착한 짝수 시간"**과 **"최초로 도착한 홀수 시간"** 딱 2가지만 기록합니다.
* 배열 형태: `visited[2][200001]` (0행은 짝수, 1행은 홀수)
* 브라운이 새로운 위치로 갈 때, **해당 홀/짝 시간에 이미 방문한 적이 있다면 큐에 넣지 않고 과감히 버립니다.**

### 💻 최적화 코드 스니펫
```python
from collections import deque

def catch_me_optimized(cony_loc, brown_loc):
    # visited[0]: 짝수 초 방문 최단 시간, visited[1]: 홀수 초 방문 최단 시간
    visited = [[-1] * 200001 for _ in range(2)]
    
    queue = deque()
    queue.append((brown_loc, 0))
    visited[0][brown_loc] = 0 # 0초(짝수)에 시작 위치 방문 처리
    
    time = 0
    while cony_loc <= 200000:
        # 1. 종료 조건: 브라운이 현재 홀짝성에 맞는 위치에 미리 도착해 기다리고 있는지 확인
        if visited[time % 2][cony_loc] != -1 and visited[time % 2][cony_loc] <= time:
            return time
        
        # 2. 브라운 이동 (현재 시간에 해당하는 노드들만 탐색)
        for _ in range(len(queue)):
            curr_pos, curr_time = queue.popleft()
            new_time = curr_time + 1
            new_parity = new_time % 2 # 다음 이동 시간의 홀/짝
            
            for next_pos in (curr_pos - 1, curr_pos + 1, curr_pos * 2):
                if 0 <= next_pos <= 200000:
                    # 핵심 가지치기: 해당 홀/짝 시간에 처음 방문하는 경우만 큐에 넣음
                    if visited[new_parity][next_pos] == -1:
                        visited[new_parity][next_pos] = new_time
                        queue.append((next_pos, new_time))
        
        # 3. 시간 증가 및 코니 이동
        time += 1
        cony_loc += time

    return -1
```
## 3. 문제 유형 판별법 (시그널 찾기)

코딩테스트에서 이와 같은 **"상태를 확장하는 BFS"** 문제를 눈치채는 키워드입니다.

1.  **"최소 시간", "가장 빨리 도착하는 방법"을 묻는다.**
    * 모든 이동의 가중치가 동일(1초)한 상태에서 최단 경로를 묻는다면 99% **BFS**입니다.
2.  **타겟(도착점)이 고정되어 있지 않고 실시간으로 변한다.**
    * 일반적인 미로 찾기와 달리 목적지가 도망가거나 맵의 장애물이 변한다면, 단순 1차원 `visited` 배열로는 풀 수 없습니다.
3.  **이동 방식에 힌트가 있다. (`+1`, `-1` 등)**
    * 그래프나 1차원 선상에서 제자리로 돌아올 수 있는 이동(와리가리)이 존재한다면, 방문 기록을 시간(초) 단위나 홀/짝 단위로 분리해야 한다는 강력한 시그널입니다.

## 4. 실전 접근 및 풀이 전략 (Thinking Process)

실전에서 움직이는 타겟 문제를 만났을 때 전개해야 할 사고의 흐름입니다.

* **Step 1. 기본 BFS 틀 스케치하기**
    * 출발점부터 시작해 `+1`, `-1`, `*2` 방향으로 퍼져나가는 기본 큐 삽입 로직을 짭니다.
* **Step 2. 'visited'의 차원(상태 공간) 고민하기 [가장 중요]**
    * *"한 번 방문한 곳을 진짜 두 번 다시 안 가도 되나?"* 를 스스로에게 묻습니다.
    * 이 문제처럼 내가 미리 도착해서 기다릴 수 있다면, 일반 `visited[X]`를 **`visited[상태][X]`** 로 확장해야 합니다. (여기서 상태는 '시간' 혹은 '홀/짝'이 됩니다.)
* **Step 3. 투-트랙(Two-Track) 판별 설계**
    * 내가 큐를 돌리며 이동하는 것(BFS)과 타겟이 조건에 따라 알아서 움직이는 것을 동시에 시뮬레이션하며 만나는 지점(조건문)을 설계합니다.
